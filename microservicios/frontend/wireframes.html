Introducción

Descripción del Sistema

AgroGestión es una plataforma web de microservicios para la gestión agrícola. Permite administrar usuarios (productores y asistentes técnicos), predios, cultivos e inspecciones fitosanitarias. Los wireframes a continuación representan el diseño de interfaz de usuario en baja fidelidad.

🏗 Arquitectura UI
Aplicación web SPA (Single Page Application) que consume 3 microservicios REST:

· MS-Usuarios (8081)
· MS-Territorial (8082)
· MS-Inspecciones (8083)
📐 Convenciones de wireframe
· Cajas grises = contenedores / secciones
· Rayado diagonal = imágenes / mapas
· Texto en mayúsculas = labels de campo
· [X] = botón de cierre / acción
· ▶ = navegación / flujo
📱 Responsividad
Cada pantalla se documenta en:

· Desktop: 1280px de ancho
· Mobile: 375px de ancho

Enfoque Mobile-First.
🔒 Autenticación
Todas las pantallas (excepto Login) requieren JWT. El token se almacena en sessionStorage y se envía en cada petición como:

Authorization: Bearer <token>

  • 1
    Panel izquierdo (desktop): Panel decorativo con descripción del sistema. Oculto en mobile para priorizar el formulario.

Anotaciones

  • 1
    Formulario wizard 3 pasos: Paso 1: datos básicos (predio, asistente, fecha, observaciones). Paso 2: plagas detectadas (múltiples). Paso 3: resultado y recomendación de tratamiento.
  • 2
    Plagas dinámicas: El usuario puede agregar N plagas. Cada plaga tiene nombre, tipo (INSECTO/HONGO/BACTERIA/VIRUS/MALEZA), severidad y % de afectación.
  • 3
    Validación inter-servicios: Al crear la inspección, el backend valida que el predio exista en MS-Territorial y el asistente en MS-Usuarios antes de guardar.
  • 4
    Sidebar de resumen (desktop): Muestra los datos del paso 1 para referencia mientras se completan los pasos 2 y 3.
  • 5
    Barra de progreso (mobile): Reemplaza el wizard de pasos horizontal por una barra de progreso numérica (1/3, 2/3, 3/3) más compacta.

Anotaciones

  • 1
    Tarjetas de plagas: Cada plaga tiene una barra visual de severidad (BAJA/MEDIA/ALTA/CRÍTICA) y borde lateral coloreado según nivel.
  • 2
    Formulario de resultado: Editable inline. Los campos consumen PUT /api/resultados/{id}. El asistente completa la recomendación y tratamiento.
  • 3
    Cambiar estado: Dropdown para cambiar entre PENDIENTE → EN_PROCESO → RESUELTO. Llama a PUT /api/inspecciones/{id}.
  • 4
    Historial: Registro de cambios de estado. En esta versión se muestra como lista estática derivada del campo FECHA_ACTUALIZACION.
  • 5
    Mobile — Botón atrás: El "◀ Insp. #001" en el header mobile navega de vuelta al listado de inspecciones con el estado preservado.
